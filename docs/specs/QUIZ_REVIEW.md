# QUIZ_REVIEW.md — クイズ問題レビュー＆修正 エージェント指示書

Claude Code のサブエージェント機能を使って、quiz.md の問題を
**レビュー → 指摘 → 修正 → 最終確認** の流れで品質担保する。

---

## エージェント構成

```
オーケストレーター
│
├── サブエージェント R（レビュアー）
│     specs/quiz.md を読み、問題・答えの誤りを洗い出して
│     specs/quiz_review.md に出力する
│
└── サブエージェント F（フィクサー）← R の完了後に起動
      specs/quiz_review.md を読み、
      specs/quiz.md を修正・上書きする
│
└── オーケストレーターが最終差分を確認してOKなら完了
```

---

## オーケストレーターへの指示（Claude Code に貼る）

```
以下の2ステップをサブエージェントで順番に実行してください。
ステップ1が完全に完了してからステップ2を開始すること。

## ステップ1 — レビュアーエージェント（サブエージェント R）

あなたは「日本語・理科・社会・算数・体育・英語に精通した
幼児教育の専門家」です。以下の観点で specs/quiz.md の
QUESTIONS 配列を1問ずつ精査し、指摘を specs/quiz_review.md に出力してください。

### レビュー観点

【正確性】
- 正解が事実として正しいか
- 不正解の選択肢が「明らかに違う」とわかるか（紛らわしすぎないか）
- 科学的・言語的に議論の余地がある問題ではないか

【難易度・適切性】
- 5歳（年長）がひらがなを読んで理解できる文章か
- 選択肢の文字数・語彙が長すぎないか
- 難しすぎる（知識として不適切）問題ではないか

【選択肢の品質】
- 3択として成立しているか（選択肢が似すぎ・バラつきすぎ）
- 正解が選択肢の番号に偏っていないか（①ばかり・②ばかりなど）
- 消去法で自明に解けてしまう問題ではないか

【表記】
- ひらがな表記が統一されているか
- 数字・単位の表記が統一されているか

### 出力フォーマット（specs/quiz_review.md）

以下の形式で出力してください。問題がない場合は「指摘なし」と記載。

---
# クイズ レビュー結果

## 要修正（must fix）
問題番号・教科・問題文を明記し、何が問題か・どう直すべきかを具体的に記載。

## 推奨修正（should fix）
軽微な表記ゆれ・難易度の微調整など。

## 正解番号の分布確認
answer:0 / answer:1 / answer:2 それぞれの件数を集計し、
偏りがあれば指摘すること。

## 総評
---


## ステップ2 — フィクサーエージェント（サブエージェント F）

あなたは「日本語・幼児教育に精通した編集者」です。
specs/quiz_review.md を読み、以下の方針で specs/quiz.md を修正してください。

### 修正方針
- 「要修正」は必ず直す
- 「推奨修正」は内容を判断した上で適用する
- 問題を削除する場合は、同じ教科の代替問題を新たに作成して補充し、
  各教科の問題数が元のままになるよう維持する
- 正解番号の偏りがある場合は選択肢の並び順を入れ替えて是正する
- 修正後も answer インデックスが選択肢配列と一致していることを必ず確認する
- QUESTIONS の総数が 100 問になっていることを確認する

修正完了後、変更した問題の番号と変更内容を
specs/quiz_fix_log.md に記録してください。
```

---

## 完了確認（オーケストレーターが手動で行う）

```bash
# レビュー結果を確認
cat specs/quiz_review.md

# 修正ログを確認
cat specs/quiz_fix_log.md

# 問題数が100問か確認
grep -c "answer:" specs/quiz.md
```

問題なければコミット：

```bash
git add specs/quiz.md specs/quiz_review.md specs/quiz_fix_log.md
git commit -m "fix: クイズ問題をレビュー・修正（マルチエージェント）"
git push
```
